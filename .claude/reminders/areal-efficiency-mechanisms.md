# AReaL Efficiency & Acceleration Mechanisms

## The Big Picture

```
Traditional RL:     [===ROLLOUT===][===TRAIN===][===SYNC===][===ROLLOUT===][===TRAIN===]
                     GPU idle       GPU idle      both idle    GPU idle       GPU idle
                     (train GPU)    (inf GPU)     (waiting)    (train GPU)    (inf GPU)

AReaL:              [===ROLLOUT========================================...]  (inference GPUs)
                         [===TRAIN===][SYNC][===TRAIN===][SYNC][===TRAIN===] (training GPUs)
                                      ↑                   ↑
                              rollout keeps going    rollout keeps going
                              (collecting next batch)
```

That overlap is the foundation. Everything else makes it tighter. There are 8 mechanisms
working together — no single trick is enough.

---

## 1. Async Rollout-Training Overlap (The Core)

**File**: `areal/infra/workflow_executor.py` — `active_submit_and_wait()`

The training GPU never waits for rollout to finish. While training step N runs, the
rollout GPUs are already generating data for step N+1 (and N+2).

```
WorkflowExecutor maintains a pipeline:
  - Continuously submits new prompts to SGLang (up to max_concurrent_rollouts)
  - Collects completed trajectories into a buffer
  - When training asks for a batch, it's already (partially) ready

prepare_batch() returns instantly if enough trajectories are buffered.
```

This is controlled by `max_concurrent_rollouts=256` — up to 256 prompts can be in-flight
simultaneously across 4 SGLang servers.

**Impact**: ~80% GPU utilization (vs 40-50% sequential)

---

## 2. Staleness Control — Don't Generate Too Far Ahead

**File**: `areal/infra/staleness_manager.py`

If rollout runs too far ahead, the data becomes stale (generated by an old policy). The
`StalenessManager` caps how far ahead:

```
max_samples = (max_head_offpolicyness + 1) × batch_size

With max_head_offpolicyness=2, batch_size=256:
  → At most 768 samples buffered (3 batches worth)
  → Data is at most 2 training steps old
```

This is the **throughput vs freshness** trade-off. Too strict = GPUs idle waiting. Too
loose = training on stale data hurts convergence.

---

## 3. NCCL Weight Sync (10-100× faster than disk)

**File**: `areal/engine/fsdp_engine.py` — `_update_weights_from_distributed()`

After each training step, new weights must reach the SGLang inference servers. Two modes:

```
Disk mode:   Actor saves to NFS → SGLang loads from NFS
             ~100-500 seconds for a large model (storage I/O bound)

XCCL mode:   Actor broadcasts via NCCL directly to SGLang GPU memory
             ~5-10 seconds (NVLink/InfiniBand speed)
             Chunked into 256MB pieces to avoid OOM
```

The config `weight_update_mode: xccl` enables the fast path. A dedicated NCCL process
group is created between training GPUs and inference GPUs during `connect_engine()`.

---

## 4. Pause/Resume — Zero Training Loss During Sync

**Files**: `areal/trainer/rl_trainer.py` (lines 458, 540),
`areal/engine/sglang_remote.py`

During weight sync, inference must pause (can't generate with half-updated weights). But
this doesn't waste time:

```
Step N training completes
  │
  ├── rollout.pause()              ← stops SUBMITTING new requests
  │                                  (in-flight requests still finish and get buffered)
  │
  ├── actor.update_weights()       ← NCCL broadcast (5-10s)
  │
  ├── _save_hf() / _evaluate()    ← checkpoint, eval happen here too
  │
  ├── rollout.resume()             ← start submitting again
  │
  └── actor.prepare_batch()        ← Step N+1 starts immediately
                                     (buffer already has trajectories from
                                      requests that completed during pause)
```

**Why no time is lost:** Pause only stops new submissions. The dispatcher's result buffer
keeps filling from requests that were already in-flight before the pause. So when
training asks for the next batch, it's partially or fully ready.

### Detailed: What Happens to In-Flight Requests During Pause

SGLang's pause (`/pause_generation`) ABORTs all in-flight requests. But abort does NOT
erase already-generated tokens:

**Case A: Abort BEFORE prefill** — request was still in queue, zero tokens generated:
```python
# sglang_remote.py line 107:
if stop_reason == "abort" and stop_message.startswith("Abort before prefill"):
    return HttpGenerationResult(output_tokens=[], ...)  # empty
```

**Case B: Abort DURING generation** — partial tokens returned:
```python
# sglang_remote.py line 115-123 (normal parsing path):
output_tokens = [x[1] for x in meta_info["output_token_logprobs"]]
# Returns [tok1, tok2, tok3] with stop_reason="abort"
```

AReaL's `agenerate()` has a retry loop that handles this:
```python
# remote_inf_engine.py line 774-834:
while stop_reason not in ["stop", "tool_calls", "length"]:
    # "abort" is NOT in this list → loop continues

    while self.workflow_executor.is_paused():
        await asyncio.sleep(0.5)     # wait for weight update

    # Re-send with accumulated tokens as prefix
    req.input_ids += gen_result.output_tokens
    result = await arequest_with_retry(...)   # continue generating
```

**Full timeline:**
```
t=0    agenerate() sends HTTP, SGLang starts generating tok1,tok2,tok3
t=3s   /pause_generation → SGLang ABORTs → returns [tok1,tok2,tok3]
t=3s   agenerate() receives partial, enters is_paused() spin-wait
t=3-8s NCCL weight update v3→v4
t=8s   /continue_generation → agenerate() wakes up
t=8s   agenerate() re-sends with input=[prompt+tok1+tok2+tok3]
       SGLang generates tok4,tok5,...tokN with NEW weights
t=18s  Final: accumulated_versions=[v3,v3,v3,v4,v4,...,v4]
```

**Tokens from different weight versions in one trajectory** are handled by:
- Per-token version stamping
- Importance weight clipping (`behave_imp_weight_cap=5.0`)
- Recomputed logprobs under current policy (`recompute_logprob: true`)

## 5. Micro-Batching with FFD Packing

**File**: `areal/utils/data.py` — `allocate_balanced_mbs()`

RL trajectories have wildly different lengths (some answers are 50 tokens, some 1024).
Naive fixed-size batching wastes GPU memory on padding:

```
Naive batching (pad to max length in batch):
  Sequence 1: [tokens tokens tokens PAD PAD PAD PAD PAD PAD PAD]  ← 70% padding
  Sequence 2: [tokens tokens tokens tokens tokens tokens PAD PAD]
  Sequence 3: [tokens PAD PAD PAD PAD PAD PAD PAD PAD PAD PAD]

FFD packing (First-Fit Decreasing):
  Micro-batch A: [long_seq_1] [short_seq_3]      ← packed to max_tokens_per_mb
  Micro-batch B: [medium_seq_2] [medium_seq_4]    ← minimal padding
  → 99% token utilization vs ~60% naive
```

The algorithm: sort sequences by length (longest first), then greedily pack each into
the first micro-batch that has room. This is controlled by `max_tokens_per_mb: 10240`
in config.

Each micro-batch goes through a separate forward pass, so the GPU processes dense data
every time instead of mostly padding.

### Detailed: FFD Algorithm Example

```
Input: 8 trajectories with token counts:
  [950, 120, 800, 200, 750, 100, 900, 300]

Step 1 — Sort descending:
  [950, 900, 800, 750, 300, 200, 120, 100]

Step 2 — Pack into micro-batches (max_tokens_per_mb=10240):
  MB-A: 950 → remaining 9290; add 900 → 8390; add 800 → 7590; add 750 → 6840
  MB-A: add 300 → 6540; add 200 → 6340; add 120 → 6220; add 100 → 6120
  All 8 fit in one MB (total 4120 < 10240)

With larger batches, multiple MBs form:
  MB-A: [seq0(950), seq6(900), seq2(800)]   pad to 950 → 3×950 = 2850 alloc
  MB-B: [seq4(750), seq7(300), seq3(200), seq1(120), seq5(100)]  pad to 750

Key: each MB pads to ITS OWN max, not the global max.
```

### Detailed: Micro-Batch Data Flow Through RL Training

**Trajectories from the same prompt CAN end up in different micro-batches.** This is fine
because advantages are pre-computed before splitting.

```
Step 1: prepare_batch() returns list of trajectories
────────────────────────────────────────────────────
  rollout_batch = [
    traj_A: {input_ids: [4, 1024], rewards: [4], ...}  ← prompt A, 4 samples
    traj_B: {input_ids: [4, 512],  rewards: [4], ...}  ← prompt B, 4 samples
    ...64 prompts total → 256 sequences
  ]

Step 2: compute_advantages() — needs group structure
────────────────────────────────────────────────────
  For traj_A (rewards=[1.0, 0.0, 1.0, 0.0]):
    mean=0.5, std=0.5
    advantages = [1.0, -1.0, 1.0, -1.0]
  MUST happen before splitting — normalization needs all 4 samples together.

Step 3: ppo_update() — concatenate ALL into one tensor
──────────────────────────────────────────────────────
  data = {input_ids: [256, global_max_len], advantages: [256, global_max_len], ...}

Step 4: FFD packing splits 256 sequences by token count
───────────────────────────────────────────────────────
  Sequences sorted by length, packed into MBs:
  MB0: [10 seqs × 950] — long sequences, minimal padding
  MB1: [15 seqs × 800] — medium sequences
  MB2: [40 seqs × 300] — short sequences, minimal padding

  Samples from prompt A may land in different MBs. This is fine —
  each sequence carries its own pre-computed advantage.

Step 5: Forward-backward loop
─────────────────────────────
  optimizer.zero_grad()
  for mb in [MB0, MB1, MB2]:
      logits = model.forward(mb)
      loss = grpo_loss_fn(logits, mb["advantages"], mb["loss_mask"])
      loss = loss * (tokens_in_mb / total_tokens)  # weight by real tokens
      loss.backward()              # gradients ACCUMULATE
  optimizer.step()                 # one update using all accumulated gradients
```

### Detailed: What One Trajectory Tensor Dict Actually Looks Like

Each `arun_episode()` returns shape `[1, seq_len]`. `GroupedRolloutWorkflow` concatenates
4 samples (different lengths) by padding to group max, then stacking on dim 0:

```
Sample 0: [1, 950] → pad to 1024 → [1, 1024]
Sample 1: [1, 820] → pad to 1024 → [1, 1024]
Sample 2: [1, 1024] → no padding → [1, 1024]
Sample 3: [1, 600] → pad to 1024 → [1, 1024]

traj_A = {
  "input_ids": [4, 1024]
    [[tok0_s0, ..., tok949_s0, PAD, ..., PAD],     ← sample 0 (950 real + 74 pad)
     [tok0_s1, ..., tok819_s1, PAD, ..., PAD],     ← sample 1 (820 real + 204 pad)
     [tok0_s2, ..., tok1023_s2],                    ← sample 2 (1024 real)
     [tok0_s3, ..., tok599_s3, PAD, ..., PAD]]     ← sample 3 (600 real + 424 pad)

  "logprobs": [4, 1024]
    prompt positions = 0.0, generation = real logprobs, pad = 0.0

  "loss_mask": [4, 1024]
    prompt = 0, generation = 1, pad = 0

  "attention_mask": [4, 1024]
    real = 1, pad = 0

  "rewards": [4]
    [1.0, 0.0, 1.0, 0.0]
}
```

### The Complete Shape Journey

```
RLVRWorkflow.arun_episode()           → {tensors: [1, seq_len]}       per sample
    ↓ (×4 via asyncio.gather)
GroupedRolloutWorkflow                → {tensors: [4, max_group_seq]}  per prompt
    ↓ (×64 prompts collected)
prepare_batch() returns               → list of 64 dicts, each [4, varying_len]
    ↓
compute_advantages()                  → same list, "advantages" added per group
    ↓ (flatten + concatenate)
ppo_update()                          → {tensors: [256, global_max_len]}
    ↓ (FFD packing)
micro-batches                         → [{tensors: [~10, 950]},     ← long seqs
                                         {tensors: [~15, 800]},
                                         {tensors: [~40, 300]}]     ← short seqs
    ↓ (forward-backward each, gradients accumulate)
optimizer.step()                      → one weight update
```

## 6. Engine Colocation — Share GPUs

**Config**:
```yaml
ref:
  scheduling_strategy:
    type: colocation
    target: actor        # ← ref runs on actor's GPUs, no extra GPUs needed
```

```
Without colocation:
  GPUs 0-3:  rollout (SGLang)
  GPUs 4-7:  actor (FSDP)
  GPUs 8-11: ref model (FSDP)     ← 4 extra GPUs just for ref!

With colocation:
  GPUs 0-3:  rollout (SGLang)
  GPUs 4-7:  actor + ref (FSDP, time-shared)   ← ref reuses actor's GPUs
```

The ref model runs its forward pass (compute ref logprobs) on the same GPUs as the
actor, just at a different time in the training loop. For GRPO with `kl_ctl=0.0`, the
ref model is skipped entirely — zero cost.

## 7. Grouped Rollout — Parallel Samples per Prompt

**File**: `areal/infra/remote_inf_engine.py` — `GroupedRolloutWorkflow`

GRPO needs `n_samples=4` completions per prompt. Instead of running them sequentially:

```python
class GroupedRolloutWorkflow(RolloutWorkflow):
    async def arun_episode(self, engine, data):
        results = await asyncio.gather(
            *[self.workflow.arun_episode(engine, data)
              for _ in range(self.group_size)]
        )
        return concat_padded_tensors(valid_results)
```

4 `agenerate()` calls happen **concurrently** via `asyncio.gather`. They don't need
separate GPUs — they're just HTTP requests landing in SGLang's continuous batching
scheduler on the same GPU pool.

### How SGLang Handles This: Continuous Batching

Traditional (static) batching:
```
Batch starts: [req_A, req_B, req_C] all generating together
  Step 50:  C finishes → C's GPU slot WASTED until A,B finish
  Step 200: B finishes → B's GPU slot WASTED until A finishes
  Step 500: A finishes → batch done, NOW accept new requests
```

Continuous batching (what SGLang does):
```
  Step 50:  C finishes → slot freed → req_D joins immediately
  Step 51:  [req_A, req_B, req_D] running
  Step 200: B finishes → slot freed → req_E joins immediately
  Step 201: [req_A, req_D, req_E] running
  GPU stays FULL the entire time.
```

Each step runs ONE forward pass for ALL requests in the batch. The KV-cache makes this
efficient — only new tokens need computation, not the full sequence.

With `n_samples=4` and `max_concurrent_rollouts=256`:
- Up to 1024 concurrent `agenerate()` calls (256 prompts × 4 samples)
- Distributed across 4 SGLang servers via round-robin
- Each server handles ~256 concurrent requests via continuous batching
- No extra GPUs needed — concurrency is at HTTP/async level

### SGLang's RadixAttention — KV-Cache Sharing

4 samples from the same prompt share prompt KV-cache:
```
Prompt: "What is 15+27?" (100 tokens)
Sample 0: [shared prompt KV-cache] → [unique generation ...]
Sample 1: [shared prompt KV-cache] → [unique generation ...]
Sample 2: [shared prompt KV-cache] → [unique generation ...]
Sample 3: [shared prompt KV-cache] → [unique generation ...]

Prompt computed ONCE, reused 4×. Saves 75% prompt computation.
```

This is why the `rid` is cached to route same-prompt requests to the same server:
```python
# remote_inf_engine.py line 756-766:
if req.rid in self.rid_to_address:
    server_addr = self.rid_to_address[req.rid]  # reuse KV-cache
```

## 8. Memory Optimizations — Fit More on Each GPU

Four complementary techniques:

### a) Gradient Checkpointing

```
Normal forward (Layer 1 → 32):
  Save ALL activations: a1, a2, ..., a32
  Memory: O(num_layers × hidden_size × seq_len) → ~8-12GB for 1.5B model

With checkpointing (save every 8th):
  Save only: a8, a16, a24, a32 (4 checkpoints instead of 32)
  Backward needs a31? → re-run forward from a24 through layers 25-31
  Memory: ~50% reduction
  Cost: ~30% of forward recomputed → 2-5% total training slowdown
```

File: `areal/engine/fsdp_engine.py` line ~858:
```python
model.gradient_checkpointing_enable(
    gradient_checkpointing_kwargs={"use_reentrant": False}
)
```

`use_reentrant=False` is the modern PyTorch approach — compatible with FSDP2 and DTensor.

### b) torch_memory_saver — Automatic GPU↔CPU Offload

**File**: `areal/utils/offload.py`

```python
torch_memory_saver.pause()    # GPU → CPU (free GPU memory)
torch_memory_saver.resume()   # CPU → GPU (restore)
```

### c) FSDP2 Sharding — Distribute Model Across GPUs

```
Full model: 3B params = ~6GB in bf16

Without FSDP (each GPU holds full copy):
  GPU0: 6GB  GPU1: 6GB  GPU2: 6GB  GPU3: 6GB  = 24GB total

With FSDP (sharded, DP=4):
  GPU0: 1.5GB  GPU1: 1.5GB  GPU2: 1.5GB  GPU3: 1.5GB  = 6GB total
  (all-gather before forward, reduce-scatter after backward)
```

### d) Multi-Tensor Gradient Operations

**File**: `areal/engine/fsdp_utils/grad.py`

```python
# Naive: Python loop, N kernel launches
for param in model.parameters():
    norm += param.grad.norm() ** 2

# Multi-tensor (TransformerEngine / Apex): single fused kernel, 3-5× faster
multi_tensor_l2norm(all_grads)
```

## The Three-Buffer Pipeline

There are three distinct buffers forming a multi-stage queue:

```
Stage 1              Stage 2              Stage 3              Stage 4
DataLoader      →    Pending Inputs  →   AsyncTaskRunner  →   Pending Results  →  Training
(prompts)            (waiting to          (in-flight on        (completed,          (consume
                      be submitted)        SGLang servers)      waiting to be        batch)
                                                                consumed)
```

### Buffer 1: `_pending_inputs` (deque)

**File**: `areal/infra/workflow_executor.py` line 287

Where task inputs wait before submission. Fed by `active_submit_and_wait()`, drained by
the producer thread when staleness capacity allows.

### Buffer 2: AsyncTaskRunner's queue (uvloop thread)

**File**: `areal/infra/async_task_runner.py`

Tasks actively being processed — `arun_episode()` running, HTTP requests in-flight.
Size controlled by `rollout.queue_size` (default: 16 × max_concurrent_rollouts).

### Buffer 3: `_pending_results` (dict)

**File**: `areal/infra/workflow_executor.py` line 288

Completed trajectories waiting for the training loop to consume them.

### How they flow together:

```
Thread 1 (Producer):  _pending_inputs → AsyncTaskRunner (when capacity > 0)
Thread 2 (uvloop):    AsyncTaskRunner processes tasks concurrently
Thread 3 (Consumer):  AsyncTaskRunner → _pending_results (notify main thread)
Main thread:          active_submit_and_wait():
                        1. Feed items into _pending_inputs (up to capacity)
                        2. wait_results() blocks until batch_size ready
                        3. Return batch to training loop

During PAUSE:
  - Producer stops feeding (paused flag)
  - Thread 2 still finishes in-flight tasks
  - Thread 3 still collects results into _pending_results
  - After RESUME: next prepare_batch() finds results already waiting
```

### Staleness Manager as back-pressure valve:

```python
capacity = min(
    max_concurrent_rollouts - running,                              # don't overwhelm SGLang
    (max_staleness + current_version + 1) * batch_size - (accepted + running)  # don't get too stale
)
```

When capacity=0, producer sleeps. Training consumes a batch → resets accepted → capacity
opens → producer wakes.

## All 8 Mechanisms Composed in One Training Step

```
TIME →

Inference GPUs (0-3):
  [...generating batch N+1 trajectories via SGLang.................]
  [collected][paused ][NCCL wt update][resume][generating batch N+2]
      ↑          ↑                                ↑
      │          │                                └── 256 concurrent rollouts (#1)
      │          └── staleness cap: max 3 batches ahead (#2)
      └── GroupedRollout: 4 samples per prompt via asyncio.gather (#7)

Training GPUs (4-7):
  [── prepare_batch (instant, buffer has data) ──]
  [── micro-batch A fwd ──][── micro-batch B fwd ──]  ← FFD packed (#5)
  [── backward (gradient checkpointing) ──────────]   ← saves 50% mem (#8a)
  [── multi-tensor grad clip ──]                      ← fused kernel (#8d)
  [── optimizer step ──]
  [── ref logp (colocated, same GPUs) ──]             ← no extra GPUs (#6)
  [── NCCL weight broadcast ──]                       ← 5-10s, not 500s (#3)
  [── save/eval (rollout paused, still draining) ──]  ← zero loss (#4)
  [── NEXT STEP: prepare_batch (buffer ready) ──]
```
